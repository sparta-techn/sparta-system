# SpartaFlow — Architecture

> Status snapshot of the **current** codebase (as built), not the target spec.
> The product is a frontend-complete prototype with a maturing backend:
>
> - **Database**: 10 migrations, ~35 tables live (identity/HR, attendance,
>   reports, projects, collaboration, notifications). See §10.
> - **Data layers**: a class-based **service** layer (`src/services`, 35
>   modules, 32 extending `BaseService`) and **repository** layer
>   (`src/repositories`, ~35 modules) exist. See §5.
> - **UI wiring**: **Auth** and **Attendance** consume live Supabase end-to-end.
>   Most other feature UIs still read localStorage-backed **mock stores** — the
>   services/repos are built but not yet wired into those screens (the pending
>   migration, §15).
>
> Doc-accuracy audit: `docs/DOCUMENTATION_STATUS.md`.

## 1. Folder Structure

```
src/
  routes/                 # TanStack Start file-based routes (URLs)
    __root.tsx            # App shell: providers + <Outlet/> + error/404 boundaries
    index.tsx             # "/" landing redirect
    unauthorized.tsx      # "/unauthorized"
    auth/                 # Public auth pages (sign-in, reset, invite, verify)
    _authenticated/       # Pathless protected layout (auth guard)
      route.tsx           # beforeLoad gate (ssr: false)
      app/                # All in-app pages, flat dot-notation routing
  features/               # Feature-first domains (see §2)
  components/
    ui/                   # shadcn/ui primitives (46 components)
    layout/               # AppShell, AppSidebar, Topbar, PageHeader
    stat-card / states / status-badge   # shared cross-feature widgets
  hooks/                  # Truly global hooks (use-mobile)
  integrations/supabase/  # Supabase clients + auth middleware + generated types
  lib/                    # theme, utils (cn), error reporting/capture
  router.tsx              # Router factory (QueryClient context)
  start.ts                # createStart: global function/request middleware
  server.ts               # Server entry
  routeTree.gen.ts        # AUTO-GENERATED — never hand-edit
  styles.css              # Tailwind v4 + design tokens
supabase/
  migrations/             # SQL schema + RLS policies + RPC functions
  config.toml
docs/                     # Specs + this architecture doc
```

Path alias: `@/*` → `src/*` (via `vite-tsconfig-paths`).

## 2. Feature Architecture

Feature-first. Each domain under `src/features/<name>/` owns its UI, state,
types, and (where present) data access. Folders are **not** uniform — they grow
only the parts they need. Observed shapes:

| Sub-path        | Purpose                                                               |
| --------------- | --------------------------------------------------------------------- |
| `components/`   | Feature-scoped React components                                       |
| `hooks/`        | Feature-scoped hooks (e.g. `attendance/hooks`, `time-tracking/hooks`) |
| `types.ts`      | Domain types                                                          |
| `store.ts`      | localStorage-backed reactive store (mock persistence)                 |
| `mock-data.ts`  | Seed data for the store                                               |
| `api.ts`        | **Real Supabase data access** (currently only `attendance`)           |
| `queries.ts`    | TanStack Query `queryOptions` factories (only `attendance`)           |
| `utils.ts`      | Pure domain helpers                                                   |
| `*-context.tsx` | Feature-level React Context (auth, analytics filters)                 |

Features present: `analytics`, `attendance`, `auth`, `checkin`, `dashboard`,
`dependencies`, `eod`, `hr`, `kanban`, `manager`, `midday`, `notifications`,
`project-analytics`, `projects`, `sprints`, `task-communication`, `tasks`,
`time-tracking`.

**Two data-access patterns coexist today:**

- **Live backend** (`auth`, `attendance`): `api.ts` calls Supabase → `queries.ts`
  wraps with TanStack Query → components consume via hooks.
- **Mock store** (everyone else): `store.ts` seeded from `mock-data.ts`, persisted
  to `localStorage`, exposed reactively through `useSyncExternalStore`. Each
  `store.ts` header explicitly states it "mirrors the future Supabase repository
  surface" so internals can be swapped for server functions without touching
  components.

## 3. Routing Architecture

**TanStack Start file-based routing** (`@tanstack/react-router`). `routeTree.gen.ts`
is generated by the router plugin — do not edit by hand.

- `__root.tsx` — single app shell. Wraps everything in providers (§7) and renders
  `<Outlet/>`. Owns `notFoundComponent` and `errorComponent` (the error boundary
  reports to Lovable via `reportLovableError`).
- `_authenticated/route.tsx` — **pathless protected layout**. `ssr: false` (the
  Supabase session lives in `localStorage`, unreadable by the server). Its
  `beforeLoad` calls `supabase.auth.getUser()` and `redirect`s to `/auth` with the
  current `location.href` as a `redirect` search param when there is no user.
- `_authenticated/app/*` — all in-app pages, using **flat dot-notation** for
  nesting (`tasks.kanban.tsx` → `/app/tasks/kanban`, `tasks.$id.tsx` → param route).
- `auth/*` — public pages: `index` (sign-in), `forgot-password`, `reset-password`,
  `accept-invitation`, `verify-email`, `session-expired`.

Router is created per-request in `router.tsx` with `{ queryClient }` as router
context and `scrollRestoration: true`. Convention reference lives in
`src/routes/README.md`.

## 4. Component Architecture

- **UI primitives** — `src/components/ui/` holds 46 shadcn/ui (new-york style,
  slate base, CSS variables) components. These are the canonical Button, Card,
  Dialog, Sheet, Table, Badge, Avatar, Tabs, etc. Never duplicate them.
- **Layout chrome** — `components/layout/`:
  - `AppShell` composes `SidebarProvider` → `AppSidebar` + `Topbar` + `<main>` +
    `FloatingActiveTimer`. Authenticated pages render their content inside `AppShell`.
  - `PageHeader`, `Topbar`, `AppSidebar`.
- **Shared widgets** — `stat-card`, `states` (empty/loading/error), `status-badge`
  at `components/` root, reused across features.
- **Feature components** — live under each feature's `components/`. Pages
  (route files) compose feature components + layout + `ui` primitives. Example:
  the dashboard route imports ~15 feature widgets (checkin, eod, midday,
  notifications, dependencies, attendance) and assembles them inside `AppShell`.

Composition over duplication is the consistent pattern: route files are thin
assemblers; logic lives in feature components/hooks/stores.

## 5. Service / Data-Access Layer

The app now has a **class-based, layered data backbone** (added after the initial
prototype). Three cooperating layers sit between components and Supabase:

```
Components / hooks / TanStack Query
      │ call
      ▼
src/repositories/*   ← domain API: aggregates + lifecycle verbs (35 repos)
      │ use
      ▼
src/services/*       ← service classes over one table/RPC, extend BaseService (35 modules)
      │ use
      ▼
src/lib/supabase + integrations/supabase   ← the single Supabase client
```

- **Service layer** (`src/services`, see `docs/SERVICES.md`) — each domain exposes
  a `<Domain>Service` extending `BaseService<Row, Insert, Update>` (generic CRUD:
  `list`, `paginate`, `getById(OrThrow)`, `create(Many)`, `update`, `upsert`,
  `remove`, `count`) plus domain methods. Shared foundation in `services/core`
  (`base-service.ts`, `client.ts` [typed `supabase` + relaxed `db`], `errors.ts`
  [`ServiceError`/`toServiceError`], `types.ts`). Import the singleton, not the class.
- **Repository layer** (`src/repositories`, see `docs/REPOSITORIES.md`) — the
  domain-facing entry point for the UI. Thin; composes one or more services and
  adds aggregate reads (`getCurrentIdentity`, `getWithMilestones`) and lifecycle
  verbs. Each takes its service(s) via constructor for testing/DI.
- **Legacy live modules** — `auth/auth-service.ts` and `attendance/api.ts` (plain
  async fns wrapping Supabase) still exist; the newer `authService` /
  `attendanceService` **compose** them rather than duplicate. `attendance/queries.ts`
  exposes `queryOptions` factories with a `attendanceKeys` hierarchy.
- **Mock stores** — each `features/*/store.ts` remains as the localStorage-backed
  stand-in for features whose UI is not yet wired to repositories. Retiring these
  in favor of repositories + TanStack Query is the pending migration (see §15).

> CLAUDE.md alignment: the mandate "all external communication goes through
> service classes; components never call APIs directly" is now **satisfied
> literally** by the service + repository layers. The remaining gap is _wiring_:
> most feature UIs still read their `store.ts`, not a repository.

## 6. Hooks Layer

- **Global**: `src/hooks/use-mobile.tsx` (breakpoint detection).
- **Feature hooks**:
  - `attendance/hooks/`: `use-today-session`, `use-timer`, `use-attendance-reminders`.
  - `time-tracking/hooks/`: `use-now`.
- **Store subscriptions**: stores expose a `useXState(selector)` hook built on
  `useSyncExternalStore` (e.g. `useTasksState`) — the reactive read path for
  mock-backed features.
- **Query hooks**: TanStack Query (`useQuery(queryOptions)`) for live features.

## 7. Context Providers

Provider tree is rooted in `__root.tsx` (`RootComponent`):

```
QueryClientProvider (TanStack Query, client from router context)
  └─ ThemeProvider (lib/theme.tsx, defaultTheme="system")
       └─ AuthProvider (features/auth/auth-context.tsx)
            ├─ <Outlet/>   ← all routes
            └─ <Toaster/>  ← sonner
```

- **AuthProvider** — bootstraps the session, subscribes to
  `supabase.auth.onAuthStateChange`, loads `profile` + `roles`, and exposes
  `AuthState` (user/profile/roles/loading + `hasRole`/`hasAnyRole`/
  `hasPermission`/`hasAnyPermission`/`refresh`/`signOut`) via `useAuth()`.
- **ThemeProvider** — light/dark/system theme.
- **Feature-scoped contexts** — `analytics/filters-context.tsx` (shared filter
  state for analytics views).

## 9. State Management

Layered, no Redux:

1. **Server/cache state** — TanStack Query (live features). QueryClient is created
   in `router.tsx` and threaded through router context.
2. **Global identity state** — React Context (`AuthProvider`).
3. **Feature shared state** — React Context (analytics filters).
4. **Mock domain state** — module-level stores + `useSyncExternalStore`, persisted
   to `localStorage` (tasks, projects, sprints, kanban, eod, midday, checkin,
   dependencies, notifications, task-communication, time-tracking).
5. **Local UI state** — `useState`/`useReducer` in components.
6. **Form state** — `react-hook-form` + `zod` resolvers.

## 10. Supabase Integration Points

`src/integrations/supabase/` (clients are generated — header says don't edit):

| File                 | Role                                                                                                                                                                                |
| -------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `client.ts`          | Browser client (publishable key). Lazy `Proxy` singleton. `localStorage` session, `persistSession`, `autoRefreshToken`. Strips bearer for opaque `sb_publishable_` keys.            |

| `types.ts`           | Generated `Database` types.                                                                                                                                                         |

## 11. Authentication Flow

1. **Bootstrap** — `AuthProvider` subscribes to `onAuthStateChange` _first_, then
   reads the existing session. Identity (`profile` + `roles`) is only refetched
   when the user id actually changes; the supabase call is deferred out of the
   auth callback (`setTimeout 0`) to avoid deadlocks.
2. **Sign-in** — `auth-service.signInWithPassword` → Supabase establishes the
   session in `localStorage`.
3. **Route gate** — `_authenticated/route.tsx` `beforeLoad` calls
   `supabase.auth.getUser()`; no user → `redirect` to `/auth?redirect=<href>`.
   Layout is `ssr: false`.
4. **Server-fn auth** — client `attachSupabaseAuth` adds the bearer; server
   `requireSupabaseAuth` validates claims and provides an RLS-scoped client.
5. **Password reset / invitation** — email link establishes a session, then
   `updatePassword` (`auth.updateUser`). Routes: `forgot-password`,
   `reset-password`, `accept-invitation`, `verify-email`, `session-expired`.

## 12. Role-Based Access Control

Two-tier, with the **database as source of truth**:

- **DB / RLS (authoritative)** — `app_role` enum: `owner`, `super_admin`, `hr`,
  `project_manager`, `team_lead`, `employee`, `viewer`. Roles stored in
  `user_roles`. SQL helper functions `has_role` / `has_any_role` drive RLS policies
  (e.g. dept/team/profile writes restricted to `hr`/`super_admin`/`owner`; role
  writes restricted to `super_admin`/`owner`).
- **Frontend (UI-gating only)** — `auth/permissions.ts` maps roles → coarse
  `Permission` keys (`users:read`, `users:write`, `roles:write`, `hr:access`,
  `owner:access`, `reports:read`, `reports:write`). `AuthProvider` precomputes
  the permission set; components gate with `hasRole`/`hasPermission` from
  `useAuth()`. `ROLE_RANK`/`ROLE_LABELS` support hierarchy/display.

The frontend matrix is explicitly documented as a _mirror of RLS intent_ — it
must never be the only enforcement point.

## 14. Coding Standards

(See `CLAUDE.md`, `docs/UI_RULES.md`, `docs/DB_RULES.md`, `.prettierrc`,
`eslint.config.js`.)

- Strict TypeScript; **no `any`**. Small components; readable naming.
- Reuse before create — search existing components/hooks/services/types first.
- Composition over duplication; never duplicate `ui/` primitives.
- All external I/O goes through a feature data module (`api.ts`/`store.ts`),
  never inline in components.
- State: prefer Context + TanStack Query + local state. No new global-state libs
  without approval.
- DB: UUID PKs, RLS on every table, never expose the service-role key client-side.
- Lazy-load routes, memoize expensive components, virtualize long lists.
- Prettier + ESLint (typescript-eslint, react-hooks, react-refresh) enforced.
- **Lovable sync**: this repo is connected to Lovable (`AGENTS.md`). Don't rewrite
  pushed history; keep the connected branch in a working state.

## 15. Architectural Improvements Needed Before Backend Integration

The single biggest gap: **most features are mock-backed**. The good news is the
stores were designed for swap-out. Recommended sequence:

1. **Wire the UI to the repository/service layers.** The boundary is now built
   (`src/repositories` → `src/services` → Supabase, §5). The remaining work is
   _consumption_: replace each feature's `store.ts` reads/writes with repository
   calls behind TanStack Query hooks, so `store.ts` becomes at most an
   optimistic/offline cache. `attendance` (`api.ts` + `queries.ts`) remains the
   reference for the query-hook shape.

2. **Replace `localStorage` stores with Supabase + TanStack Query.** Each
   `store.ts` read/write maps to a query/mutation. Because components consume
   stores via `useXState` selectors, introduce equivalent query hooks so the
   component layer changes minimally. Retire `mock-data.ts` seeds (move any needed
   seeds to SQL migrations / fixtures).

3. **Land the remaining schema.** Most domains now have tables + RLS + indexes
   (identity/HR, attendance, reports, projects, collaboration, notifications — §10).
   Still missing: **`tasks`, `comments`, `sprints`** (their services/repos are
   wired and waiting), plus kanban, time-tracking, files, and analytics
   aggregates. Derive them from the existing feature `types.ts` and the specs in
   `docs/DATABASE_DESIGN.md` / `docs/DatabaseSchema.md`.

4. **Decide read path: server functions vs. direct client queries.** The
   infrastructure for authenticated server functions exists
   (`requireSupabaseAuth` + `attachSupabaseAuth`) but is currently unused by
   feature code (attendance queries the client directly). Choose deliberately:
   direct client+RLS for simple reads, server functions for privileged/cross-row
   logic. Document the rule.

5. **Harden RBAC parity.** Keep `permissions.ts` in lockstep with RLS as new
   tables land; add a check (test or generated assertion) so the UI matrix can't
   silently drift from policy. Wire `owner:access`/`hr:access` gates into the
   sidebar/routes (some pages are role-sensitive but not yet guarded beyond auth).

6. **Add route-level data loading & code-splitting.** Move from
   component-mounted fetching toward route `loader`s using
   `queryClient.ensureQueryData(queryOptions)` for the live features, and lazy-load
   heavy routes (analytics/charts) to honor the performance standard.

7. **Input validation on writes.** `zod` is present for auth/forms; extend schemas
   to every mutation boundary before it hits Supabase.

8. **Realtime & files.** Two specced capabilities with no integration yet —
   plan Supabase Realtime subscriptions (notifications, kanban, presence) and
   Supabase Storage (files/attachments, avatars) as explicit milestones.

9. **Error/empty/loading consistency.** A `states` module exists; ensure every
   newly-wired query renders through it (and surfaces errors to
   `reportLovableError`) instead of bespoke handling.

10. **Generated-file discipline.** `client.ts`, `client.server.ts`,
    `auth-middleware.ts`, `types.ts`, and `routeTree.gen.ts` are generated —
    regenerate `types.ts` from the live schema after each migration rather than
    editing by hand.
