# Audit System — SpartaFlow Hub

Every sensitive action leaves an immutable, queryable trail. Used by HR, Owner, Super Admin, security incident response, and (when needed) external auditors.

## 1. Storage

`audit_logs` is append-only:

- `id bigint identity pk`
- `actor_id uuid` — `auth.uid()` at time of action; null for system jobs.
- `actor_role app_role` — captured at write time so demotions don't rewrite history.
- `action audit_action` — enum.
- `target_table text`, `target_id uuid` — what was acted on.
- `before jsonb`, `after jsonb`, `diff jsonb` — pre/post snapshots and computed delta.
- `ip inet`, `user_agent text` — populated from request context via `set_config`.
- `correlation_id uuid` — propagated end-to-end from the request.
- `occurred_at timestamptz` — DB clock.

Hardening:
- `UPDATE` and `DELETE` grants revoked from every role.
- BEFORE UPDATE/DELETE trigger `tg_prevent_audit_mutation` raises `feature_not_supported`.
- Partitioned by month (`PARTITION BY RANGE (occurred_at)`) once volume warrants — keeps indexes small.
- BRIN index on `occurred_at` for cheap time-range scans.

## 2. What Gets Audited

| Domain | Events |
|---|---|
| Auth | sign-in, sign-out, failed sign-in (rate-limited), password change, MFA enroll/disable. |
| Identity | profile updates (any field), employee_profiles changes. |
| Roles | role grant / revoke (`user_roles` insert/update). |
| Attendance | override, manual edit, auto-close. |
| Workflow | corrections to morning/midday/eod (admin path). |
| Leaves | request, approve, reject, cancel. |
| Dependencies | create, status transitions, escalate, cancel, reassign. |
| Announcements | publish company-wide, edit after publish, delete. |
| Settings | any `settings` change, `working_rules` change. |
| Integrations | connect, disconnect, token rotate, webhook signature failure. |
| Storage | privileged downloads of `documents` / `exports`. |
| Admin | impersonation, role assignment via admin UI, offboarding. |

System events (no `actor_id`):
- pg_cron job runs that wrote to audited tables (e.g. `close_open_attendance`).
- Outbox dispatcher actions.
- Auth Admin API calls from Edge Functions (actor_id = the human who triggered the Edge Function).

## 3. How Audits Are Written

Two write paths:

1. **Trigger-based** (`tg_write_audit_log`) — generic AFTER INSERT/UPDATE/DELETE on each audited table. Diffs OLD vs NEW and writes one row.
2. **Function-based** — privileged functions (`grant_role`, `override_attendance`, `offboard_user`, etc.) write a richer audit row themselves with a typed `action` and human-readable context that a generic diff cannot capture.

Request context (`actor_id`, `ip`, `user_agent`, `correlation_id`) is injected per request:

- Server fns / Edge Functions call `set_config('request.actor_id', ...)`, `set_config('request.ip', ...)`, etc., before any audited statement.
- The trigger reads these via `current_setting('request.xxx', true)`.
- Missing values are tolerated (recorded as null) — never block the write.

## 4. RLS

- `audit_logs.SELECT` — `using (current_perm('admin.audit.read'))`. Granted to Owner, Super Admin, HR.
- No INSERT policy for `authenticated`; writes happen via trigger or SECURITY DEFINER function under `service_role`-equivalent privileges.
- HR cannot see audit rows targeting `integration_accounts` (filter in the policy by `target_table not in ('integration_accounts','integration_links')`) — those are Owner/Super Admin only.

## 5. Querying & UI

- `/admin/audit` page provides filters: actor, target user, target table, action, date range, correlation_id.
- Backed by an indexed query plan: `(target_table, target_id, occurred_at desc)` is the workhorse index.
- Detail view shows `before` / `after` side-by-side with a `diff` highlight.
- Exports (CSV/PDF) generated by a server fn that writes to `exports/` bucket and is itself audited.

## 6. Retention

| Class | Minimum retention | Storage |
|---|---|---|
| All audit rows | 1 year | hot Postgres partition |
| Role grants/revokes, offboarding, integration changes | 3 years | hot + cold export monthly |
| Failed sign-in / security events | 1 year | hot |

Retention enforced by a monthly job that drops the oldest partition after exporting it to `exports/audit/YYYY-MM.parquet` for cold storage.

## 7. Correlation

Every request gets a `correlation_id` (UUID):

- Generated at the edge of the request (TanStack server fn middleware / Edge Function).
- Propagated to `set_config('request.correlation_id')` for DB triggers.
- Emitted in Sentry, server logs, audit rows, outbox events.
- Surfaced in admin UI so a single user complaint can be traced across logs, traces, and audit history.

## 8. Tamper Evidence

- Append-only enforcement (grants + trigger).
- Optional: monthly hash chain — each partition root hash signed and stored externally. Detects tampering by anyone with raw DB access.
- All schema changes to `audit_logs` themselves are reviewed by two engineers and recorded in their own migration audit.

## 9. Privacy

- `before`/`after` JSON respects `redacted_columns` config — sensitive columns (salary band, government IDs) are stored as `"<redacted>"`.
- GDPR erasure: PII fields in `before`/`after` are tombstoned with `"<erased>"` while keeping the action/timestamp/actor — auditability of "the deletion happened" is preserved.
